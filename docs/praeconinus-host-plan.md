# Praeconinus Host Configuration Plan

Praeconinus is a Raspberry Pi 4 (aarch64-linux) running NixOS, serving as a guest-facing wifi portal accessible via Tailscale Funnel at `https://praeconinus.neko-bicolor.ts.net`.

## Purpose

Allow guests who aren't on the local network to use their cellular connection to configure wifi credentials via the homelab UI/API, exposed over Tailscale Funnel on a single domain with sub-path routing (`/api`, `/ui`).

## Public URL

`https://praeconinus.neko-bicolor.ts.net`

- `/ui` → static files (homelab-ui Astro build)
- `/api` → reverse proxy to homelab-api (localhost:35427)

      inputs.nixvim.nixosModules.nixvim
      inputs.sops-nix.nixosModules.sops

      inputs.self.modules.nixos.system-base
      inputs.self.modules.nixos.caddy

      # caddy config for sub-path routing, and homelab-api-service directly
      inputs.self.modules.nixos.homelab-api-service

      ./../_private/praeconinus/hardware.nix
      ./../_private/praeconinus/system.nix
      ./../_private/praeconinus/security.nix

Key differences from augustus:

- `aarch64-linux` system
- No postgres, kanidm, minecraft, restic, etc.
- No `homelab-api` complete module (which bundles caddy vhost + secret-provisioner for the domain-based setup). Instead, use `homelab-api-service` directly and write a custom Caddy config for sub-path routing.
- No `homelab-ui` complete module — Caddy serves static files at `/ui` sub-path directly in the programs/caddy config.

### 4. `nix-config/modules/_private/praeconinus/security.nix`

```nix
{ inputs, lib, config, pkgs, ... }:
{
  security.sudo.execWheelOnly = true;

  # Trust internal CA for HTTPS verification
  security.pki.certificateFiles = [ (inputs.self + "/certs/alford-root.crt") ];

  networking.firewall = {
    enable = true;

    # Tailscale interface: only 443 (for Funnel)
    interfaces.tailscale0 = {
      allowedTCPPorts = [ 443 ];
    };

    # LAN: only SSH on port 22
    allowedTCPPorts = [ 22 ];
  };

  # Don't open SSH on tailscale (handled above — only 443 on ts)
  services.openssh.openFirewall = false;

  # NO ssh-cert-renewer — this host cannot reach the CA.
  # But we still trust the SSH User CA cert for inbound connections
  # (handled by the system-base openssh module's TrustedUserCAKeys).
  #
  # The host cert at /etc/ssh/ssh_host_ed25519_key-cert.pub will need
  # to be manually provisioned (one-time sign via step-cli from a
  # machine that can reach the CA) or left unsigned.
}
```

**Note on SSH:** The shared `openssh` module already sets `TrustedUserCAKeys` pointing to the SSH user CA public key (`certs/ssh_user_ca_key.pub`). This means any SSH client presenting a cert signed by the user CA will be trusted — which is exactly what we want. The host just won't auto-renew its own host cert since it can't reach step-ca.

We need to override the openssh `extraConfig` to remove the `HostCertificate` line if no host cert will be present, or manually provision one with a long lifetime:

```nix
# Override to remove HostCertificate if no cert is provisioned
services.openssh.extraConfig = lib.mkForce "";
```

### 7. `nix-config/modules/_private/praeconinus/programs.nix`

```nix
{ lib, config, pkgs, inputs, ... }:
let
  uiPkg = inputs.self.packages.aarch64-linux.homelab-ui-praeconinus;
  apiPkg = inputs.self.packages.aarch64-linux.homelab-api;
in
{
  programs.git.enable = true;
  programs.zsh.enable = true;

  environment.systemPackages = with pkgs; [ vim ];

  # --- homelab-api configuration ---
  services.homelab-api = {
    enable = true;
    pkg = apiPkg;
    originUrl = "https://praeconinus.neko-bicolor.ts.net/api";
    privateKeySecretPath = config.sops.secrets.homelab_api_jwk_password.path;
    provisionJwks = true;
    apiKeys = {}; # No API keys for this deployment
    featureFlags = [
      "Config_Wifi.create"
      "Config_Certs.view"
    ];
  };

  # --- Caddy: sub-path routing on Tailscale Funnel ---
  services.caddy.virtualHosts."praeconinus.neko-bicolor.ts.net" = {
    extraConfig = ''
      handle_path /api/* {
        reverse_proxy 127.0.0.1:35427
      }

      handle_path /ui/* {
        root * ${uiPkg}
        header /*.html Cache-Control "no-cache, must-revalidate"
        file_server
      }

      # Root redirect to UI
      handle {
        redir / /ui/ permanent
      }
    '';
  };

  # --- Tailscale Funnel ---
  # Funnel must be configured via `tailscale funnel 443` on the host
  # or via tailscale serve configuration. Caddy listens on 443 and
  # Tailscale Funnel proxies external traffic to it.
}
```

## Tailscale Funnel Setup

After deployment, on the host:

```bash
tailscale funnel --bg 443
```

Or configure via `tailscale serve`:

```bash
tailscale serve --bg --https=443 http://localhost:443
tailscale funnel --bg 443
```

This makes `https://praeconinus.neko-bicolor.ts.net` publicly reachable, with Tailscale handling TLS termination and forwarding to Caddy on port 443.

- No other ports exposed

## SSH Configuration Summary

- **Inbound auth:** Trusts SSH user certs signed by `certs/ssh_user_ca_key.pub` (inherited from `system-base` → `openssh` module)
- **Host cert:** Not auto-renewed (no CA connectivity). Either:
  - Manually provision a long-lived host cert via step-cli from another machine
  - Or omit host cert (clients connect without verifying host via cert)
- **ssh-cert-renewer:** Disabled (not called in module list, no sops secret for JWK password)
- **Fallback:** Public key auth remains available for jacob

## Open Questions / Decisions

1. **homelab-secret-provisioner**: Augustus uses it to generate JWKs and HMAC. Praeconinus needs JWKs too. Either:
   - Include `homelab-secret-provisioner` module (requires its sops secret)
   - Or provision JWKs manually / at first boot

2. **OIDC**: The homelab-api env template references `KANIDM_OPENID_PROVIDER_URL`. If no OIDC is needed for guest usage, this could be left as-is (feature flags restrict to wifi+certs only) or the env template may need a conditional.

3. **Caddy TLS on Tailscale Funnel**: When using Funnel, Tailscale terminates TLS externally. Caddy may need to listen on HTTP internally (not HTTPS) since Funnel forwards decrypted traffic. Confirm Caddy should bind `:80` or if Tailscale serves traffic directly to Caddy's HTTPS port.

4. **aarch64 cross-compilation**: If the build host is x86_64, ensure the flake supports cross-compilation to aarch64-linux or that builds are performed on the Pi itself.
